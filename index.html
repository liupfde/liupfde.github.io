二、重要的变量

  • ctl

    ctl是一个AtomicInteger原子类对象。一个ctl变量包含两部分信息:线程池的运行状态(rs)和线程池内有效线程的数量(wc)。由于一个int型的变量是由32位二进制数组成，所以ctl的高3位表示线程池的运行状态，用低29位来表示线程池内有效线程的数量(所以线程池的线程数量的上限就是29个1表示的数字）。而ctl的数值就是由这两个变量计算得出。(ctlOf(int rs, int wc)函数)

    同样的如果知道了ctl值也可以通过以下两个方法计算得出线程池的运行状态和线程池内有效线程的数量。

    其中CAPACITY表示高三位是0，低29位是1的一个int型的数。

  • 线程池的运行状态(5种)

    1. RUNNING(运行状态)：能接受新提交的任务并且也能处理阻塞队列中的任务。
    2. SHUTDOWN(关闭状态)：不再接受新提交的任务但是可以继续处理阻塞队列中保存的任务。在线程池处于RUNNING状态时调用shutdown()方法可以进入该状态。
    3. STOP：不能接受新提交的任务也不能处理阻塞队列中已保存的任务，并且会中断正在处理中的任务。在线程池处于RUNNING或者SHUTDOWN状态时调用showdownNow()方法可以进入此状态。
    4. TIDYING(清理状态)：所有的任务都已终止了，workCount(有效线程数)为0，线程进入该状态后会调用terminated()方法以让该线程池进入TERMINATED状态。当线程池处于SHUTDOWN状态时，如果此后线程池内没有线程了并且阻塞队列内也没有待执行的任务了，线程就会进入该状态；当线程处于STOP状态时，如果此后线程池内没有线程就会进入到该状态。
    5. TERMINATED：terminated()方法执行完就进入该状态。

    上图是五种状态的数值。例如 当rs < SHUTDOWN，那么表示线程池处于RUNNING状态。

三、构造方法中的重要参数

根本的构造方法(所有的构造方法最终都会调用到这一个)：

  1. corePoolSize

    表示线程池中一直存活着的线程的最小数量，也被称作核心线程。默认情况下，核心线程是按照需要创建并启动的，也就是说，只有当线程池接收到我们提交给他的任务后，他才会创建并启动一定数量的核心线程来执行这些任务。也可以通过prestartCoreThread()或者prestartAllCoreThreads()方法是的新任务还未提交时线程池就已经创建并启动课一个或所有的核心线程。

  2. maximumPoolSize

    maximumPoolSize 表示线程池内能够容纳线程数量的最大值。但是不能超过常量CAPACITY的数值大小。

    当通过execute(Runnable)方法提交一个任务到线程池时，如果线程池处于运行状态(RUNNING)的线程数量少于核心线程数(corePoolSize)，那么即使有一些非核心线程处于空现状态，系统也会倾向于创建一个新的线程来执行任务。如果此时处于运行状态(RUNNING)的线程数量大于核心线程数(corePoolSize), 但又小于最大线程数(maximumPoolSize), 那么系统将会去判断线程池内部的阻塞队列 workQueue 中是否还有空位子. 如果发现有空位子, 系统就会将该任务先存入该阻塞队列； 如果发现队列中已没有空位子(即: 队列已满), 系统就会新创建一个线程来执行该任务；如果corePoolSize == maximumPoolSize，则该线程池容量是固定的。

  3. keepAliveTime

    keepAliveTime表示空闲线程处于等待状态的超市时间(超过该时间线程停止工作)。当总线程数大于corePoolSize且allowCoreThreadTimeOut为false时，这些多出来的非核心线程一旦进入空闲状态就开始计算各自的等待时间，一旦等待时间超过keepAliveTime，该线程就会停止工作(terminated)，而核心线程不会受超时时间的限制。但是如果allowCoreThreadTimeOut为true，则核心线程也会收到限制。

  4. workQueue

    阻塞队列(BlockingQueue<Runnable>)。基于”生产者-消费者”模型。队列对新提交的任务的三种处理策略：

    • 直接切换

      常用的队列是SynchronousQueue(同步队列)。这种队列内部不会存储元素，每一次插入操作都会先进入阻塞状态，直到另一个线程执行了删除操作，然后插入操作才会执行。同样的每一次删除操作也会等待插入操作的执行完毕。当提交一个任务到使用这种阻塞队列的线程池后，线程池会检测是否有可用的空间线程来执行该任务，如果没有就直接创建一个线程而不是将该任务放到阻塞队列中。这种策略的目的是让每一个提交的任务都能得到立即执行。这要求线程池的最大值是线程池允许的最大值Integer.MAX_VALUE。静态工厂方法Executors.newCachedThreadPool()使用这种队列。

    • 无界队列

      如基于链表的阻塞队列LinkedBlockingQueue。这种情况下maximumPoolSize起不到作用，因为当线程数大于corePoolSize时总会向无界队列中添加新的线程，直到达到线程池所允许的最大值。静态工厂方法Executors.newFixedThreadPool()使用这种队列

    • 有界队列

      如ArrayBlockingQueue。可以将线程池最大的线程数maximumPoolSize限定在某个值以内，并通过队列来降低资源的消耗。

  5. threadFactory

    线程工厂，用于创建线程。如果在创建线程池的时候没有指定threadFactory参数，则默认使用Executors.defaultThreadFactory()方法创建线程工厂。如果想为线程工厂设置参数如名字、优先级那么就需要自己去实现ThreadFactory接口，并实现其抽象方法newThread()。

  6. handler

    线程池拒绝策略。以下条件满足其中一个就会触发拒绝策略。

    • 线程池处于SHUTDOWN状态(不论线程池和阻塞队列是否已满)。
    • 线程池所有的线程处于运行状态且线程池中的阻塞队列已满。

    拒绝策略的方式:

    1. AbortPolicy

      抛出异常

    2. CallerRunsPolicy

      将新提交的任务放到ThreadPoolExecutor.execute()方法所在的线程中执行

    3. DiscardPolicy

      不执行新提交的任务

    4. DiscardOldestPolicy

      • 当线程已关闭就不执行这个任务，和DiscardPolicy策略一样
      • 当线程未关闭，会将阻塞队列中处于队首(head)的那个任务从队列中移除，然后再将这个新提交的任务加入到该阻塞队列的队尾(tail)执行。

五、线程池的关闭

线程池有两种关闭方式:shutdown和shutdownNow

  • shutdown()

    调用该方法会将线程池的状态变为SHUTDOWN状态，此时不能再向线程池中添加任务，否则将会抛出RejectedExecutionException异常。但是线程池要等待池内所有任务结束之后才会退出。

  • shutdownNow

    调用该方法后线程池立即结束，并且停止正在执行的任务，不再执行线程池中的还在等待的任务。

一、EXPLAIN信息详解

字段解释

  • id

    表示执行查找或操作表的顺序。在执行操作的时候会出现以下三种情况

    1. id值相同 执行的顺序为由上至下

    2. id值不同，则id值越大的越先被执行

    3. id值有的相同有的不同，这种情况下id值相同的从上向下进行，id不同的id值越大优先级越高

  • select_type

    查询的类型，用于区分普通查询、联合查询、子查询等复杂查询。包括一下类型:

    • SIMPLE: 简单的select查询，查询中不包含子查询或者union
    • PRIMARY: 查询中包含很多复杂的子部分，最外层的查询就被标记为primary
    • SUBQUERY: 在select或where列表中包含了子查询
    • DERIVED: 在FROM列表中包含的子查询被标记为derived
    • UNION: 如果第二个select出现在union之后，则被标记为union；如果union包含在from子句的子查询中，则外层select被标记为derived
    • UNION RESULT: 从union表获取结果的select

  • type

    访问类型。SQL查询优化的重要指标，结果从好到坏依次是：

    system > const > eq_ref > ref > fulltext > ref_or_null > index_merge > unique_subquery > index_subquery > range > index > ALL

    一般来说，好的sql查询至少达到range级别，最好能达到ref

    • system

      表只有一行记录(等于系统表)，是const类型的特例，一般不会出现

    • const

      表示通过索引一次就找到了，const用于比较primary key或者unique索引。因为只需要匹配一行数据所以很快。如果将主键置于where列表中，mysql就能将该查询转换成一个const

    • eq_ref

      唯一性索引扫描，对于每个索引键，表中只有一条记录与之匹配。常见于主键索引或者唯一索引。

    • ref

      非唯一性索引扫描，返回匹配某个单独值的所有行。

    • range

      只检索给定范围的行，使用一个索引来选择行。key列显示使用了哪个索引。一般就是在where语句中出现了between、<、>、in的查询。比全表扫面好。

    • index

      表示遍历扫描索引树，读全表，但是是从索引中读取所以比ALL快。

    • ALL

      遍历全表。读硬盘。慢。

  • possible_keys

    列出查询涉及到的字段上存在的索引，但是该索引不一定在实际的查询中被使用。

  • key

    实际使用的索引，如果为NULL，则表示没有使用索引。

  • key_len

    表示索引的最大的可能字节数，并不是实际使用索引的字节数。长度越短越好。

  • ref

    显示索引的哪一列被使用了，可能是一个常量const。

  • rows

    根据表统计信息及索引的选用情况，大致估算出找到所需记录需要读取的行数。

  • Extra

    不在其他字段中显示，但是比较重要的额外信息

###问题 : MySQL多表联查效率高还是多次单表查询效率高?

在业务中碰到一个情景，需要结合两张不同的表中查出数据。例如首先要从A表中判断要查的数据id有没有，如果存在则再从B表中查处具体的数据。这时就可以采用问题中所描述的两种方案。

回答:

​ 使用多次单表查询。原因如下:

  1. 不利于写操作。有的操作执行读数据时，会锁住被读的数据，阻塞其他业务对该部分的更新操作。如果采用JOIN涉及多个聚合函数，则相当于同时锁住了多张表，不能进行读写，影响了系统的整体性能。

    (注:有可能会碰到第一步查询时数据量过大的情况，这时或许可以对第一步进行分页查询，然后第二步中从第一步的结果中再去查。)

  2. 不利于维护。如果业务发生变动，某一张表的字段改了，那么该SQL语句也会失效。即耦合度太高。

ASCII、Unicode、UTF-8之间的区别和联系

发表于 2019-04-19

1. ASCII

计算机的内部，所有的信息最终都表示为一个二进制的字符串。每一个二进制位(bit)有0和1两种状态，因此一个字节(8bit)可以组合出256种状态。ASCII码是美国制定的表示英语字符与二进制位之间关系的一套字符编码。ASCII一共规定了128个字符编码，如空格”SPACE”是32(二进制00100000)，大写字母A是65(二进制01000001)。这128个符号，只占用一个字节的后7位，最前面的1位统一规定为0。

2. 非ASCII码

英语用128个符号编码就够了，但是用来表示其他语言，128个符号是不够的。比如，在法语中，字母上方有注音符号，它就无法用ASCII码表示。于是，一些欧洲国家就决定，利用字节中闲置的最高位编入新的符号。比如，法语中的é的编码为130(二进制10000010)。这样一来，这些欧洲国家使用的编码体系，可以表示最多256个符号。 但是，这里又出现了新的问题。不同的国家有不同的字母，因此，哪怕它们都使用256个符号的编码方式，代表的字母却不一样。比如，130在法语编码中代表了é，在希伯来语编码中却代表了字母Gimel (ג)，在俄语编码中又会代表另一个符号。但是不管怎样，所有这些编码方式中，0—127表示的符号是一样的，不一样的只是128—255的这一段。 至于亚洲国家的文字，使用的符号就更多了，汉字就多达10万左右。一个字节只能表示256种符号，肯定是不够的，就必须使用多个字节表达一个符号。比如，简体中文常见的编码方式是GB2312，使用两个字节表示一个汉字，所以理论上最多可以表示256x256=65536个符号。 中文编码的问题需要专文讨论，这篇笔记不涉及。这里只指出，虽然都是用多个字节表示一个符号，但是GB类的汉字编码与后文的Unicode和UTF-8是毫无关系的。

3. Unicode

Unicode是一种统一编码的方式，现在最多可以容纳100多万个符号。

Unicode的问题

  • Unicode只是一个符号集，他只规定了符号的二进制代码，却没有规定这个二进制代码如何存储

  • 每个字符的长度不同，有的需要两个字节比如汉字，而有的字符甚至需要三个或者四个字节来表示。这里又产生一下两个问题

    1. 如何区别ascii码和unicode码。计算机怎么知道三个字节表示一个符号而不是表示三个符号

    2. 有的字符例如英文字符只需要一个字节表示，如果Unicode统一规定，每个符号用三个或者四个字节表示，那么必然造成存储的浪费。

以上问题造成的结果就是出现了许多种不同的二进制格式，可以用来表示unicode。

4. UTF-8

UTF-8就是在互联网上使用最广的一种unicode的实现方式。其他实现方式还包括UTF-16和UTF-32。这里的关系是，UTF-8是Unicode的实现方式之一。 UTF-8的编码规则很简单，只有二条： 1）对于单字节的符号，字节的第一位设为0，后面7位为这个符号的unicode码。因此对于英语字母，UTF-8编码和ASCII码是相同的。 2）对于n字节的符号（n>1），第一个字节的前n位都设为1，第n+1位设为0，后面字节的前两位一律设为10。剩下的没有提及的二进制位，全部为这个符号的unicode码。

  • UTF-8的特点

    使用变长的编码方式。他可以使用1～4个字节表示一个字符，根据不同的符号变化字节的长度。

  • UTF-8的编码规则

    1. 对于单字节的符号，字节的第一位设置为0，后面7位为这个符号的unicode码。因此对于英语字母，UTF-8编码和ASCII编码是相同的。
    2. 对于n字节的符号(n>1)，第一个字节前n位设为1，第n+1位设为0，后面字节的前两位一律设为10。剩下没有提及的二进制位，全部为这个符号的unicode码。

b+树的基本原理与索引

发表于 2019-04-18

为了描述B-Tree，首先定义一条数据记录为一个二元组[key, data]，key为记录的键值，对于不同的数据记录key是互不相同的；data为数据记录除key以外的数据。则B-Tree是满足下列条件的数据结构:

  • d为大于1的一个正整数，称为B-Tree的度。
  • h为一个正整数，称为B-Tree的高度
  • 每个非叶子结点有n-1个key和n个指针组成，其中d<=n<=2d。
  • 每个叶子结点最少包含一个key和两个指针，最多包含2d-1个key和2d个指针，叶结点的指针为null
  • 所有的叶结点有相同的深度，等于树高h
  • key和指针互相间隔，节点两端是指针
  • 如果某个指针在节点node最右边且不为null，则其指向节点的所有key大于v(keym)，其中v(keym)为node的最后一个key的值。
  • 如果某个指针在节点node耳朵左右相邻key分别是key(i)和key(i+1)且不为null，则其指向节点的所有key:v(key(i)) < k <v(key(i+1)) 下图是一个d=2的B-Tree示意图

  • 一个节点由以下三部分组成

    • key
    • data
    • 指针

有以上性质可以定义出B-Tree的查找算法:

​ 首先从根节点进行二分查找，如果找到就返回对应的节点data，否则对相应区间的指针指向的节点递归进行查找，直到找到节点(查找成功)或找到null指针(查找失败)。

B-Tree的一些性质:

  • 一个度为d的B-Tree，设其索引N个key，则其树高h的上限$log_d(N+1)/2$，检索一个key的时间复杂度为O($log_dN$)。

B+Tree

与B-Tree相比，B+Tree有以下的不同点:

  1. 每个节点的指针上限为2d而不是2d+1。

  2. 内节点不存储data，只存储key；叶子节点不存储指针。

    下图是一个B+Tree的示意图

    1. 由于并不是所有节点都具有相同的域，因此B+Tree中叶节点和内节点一般大小不同。这点与B-Tree不同，虽然B-Tree中不同节点存放的key和指针可能数量不一致，但是每个节点的域和上限是一致的，所以在实现中B-Tree往往对每个节点申请同等大小的空间。

带有顺序访问指针的B+Tree

在B+Tree的每个叶子节点增加一个指向相邻叶子节点的指针，就形成了带有顺序访问指针的B+Tree。做这个优化的目的是为了提高区间访问的性能，例如上图如果要查询key为从18到49的所有数据记录，当找到18后，只需顺着节点和指针顺序遍历就可以一次性访问到所有数据节点，极大提到了区间查询效率。

B树与索引

一般来说，索引本身很大，不可能全部存储在内存中，因此索引往往以索引文件的形式存储在磁盘上。这样索引查找的过程就要产生磁盘I/O的消耗，I/O的消耗要比内存高几个数量级，所以要尽量减少查找过程中磁盘I/O的存取次数。

根据局部性原理和磁盘预读原理一次I/O的时候会预读页(page)的整数倍的数据，而数据库将一个节点的大小设置等于一个页，这样每个节点只需要一次磁盘I/O就可以完全载入。

MySQL索引的实现

  • MyISAM引擎

    MyISAM采用非聚簇索引的形式，即叶结点的data域存放的是数据记录的地址。MyISAM的索引文件仅仅保存数据记录的地址。

    在MyISAM中，主索引和辅助索引在结构上没有区别，只是主索引要求key是唯一的。

  • InnoDB引擎

    InnoDB采用聚簇索引的形式，即InnoDB的数据文件本身就是索引文件。对于主键索引来说，其中key是索引，data域存放的就是数据。

    对于辅助索引来说，key是辅助索引，而data域存放的是主键，最终会通过主键查找出具体的数据。
